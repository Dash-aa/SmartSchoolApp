# Linting у проєкті SmartSchool

## Обраний лінтер та причини вибору

Для проєкту SmartSchool було обрано інструмент статичного аналізу коду **Checkstyle**.

Checkstyle використовується для перевірки стилю коду в Java-проєктах і дозволяє забезпечити єдині стандарти оформлення.

Основні причини вибору:
- інтеграція з Maven;
- простота налаштування через файл `checkstyle.xml`;
- контроль стилю коду;
- виявлення типових помилок;
- покращення читабельності та підтримуваності коду.

Додатково у проєкті використовуються:
- **SpotBugs** — для виявлення потенційних помилок у коді;
- **Spotless** — для автоматичного форматування коду.

## Базові правила та їх пояснення

У рамках лабораторної роботи було створено конфігураційний файл **checkstyle.xml**, який містить основні правила перевірки.

Найважливіші правила:

- **UnusedImports**  
  Забороняє невикористані імпорти, що зменшує "захаращення" коду.

- **AvoidStarImport**  
  Забороняє використання `*` в імпортах, що покращує читабельність.

- **NeedBraces**  
  Вимагає використання фігурних дужок у конструкціях if/else/for.

- **WhitespaceAfter / NoWhitespaceBefore**  
  Контролює правильне використання пробілів.

- **RedundantImport**  
  Забороняє дублювання імпортів.

- **EmptyBlock**  
  Забороняє порожні блоки коду.

- **EqualsHashCode**  
  Перевіряє, щоб методи `equals()` і `hashCode()` були узгоджені.

- **HiddenField**  
  Забороняє приховування полів класу параметрами методів.

- **LineLength**  
  Обмежує довжину рядка до 120 символів.

- **FileTabCharacter**  
  Забороняє використання табуляції.

Ці правила допомагають зробити код більш чистим, зрозумілим і однаковим у всьому проєкті.
